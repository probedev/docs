## API Endpoint

Every request is sent to our API endpoint using either the GET or POST method. All responses are formatted using JSON and follow a specific response message format

```
https://api.probe.dev/v1/probe/file
```

###  API Call Methods: GET or POST 

Here are the API call methods we currenlty support. This covers all of the most popular methods of passing parameters over http, so you can use what they are more comfortable with.

- GET request (parameters are included as part of url) For example:

  ```powershell
  curl -X GET https://api.probe.dev/v1/probe/file?token=developer-token&only=ffprobe
  ```
- POST request with three possible Content-Type headers Parameter format is the same as for GET, but parameters are included inside the POST body:
  - multipart/form-data
  - application/x-www-form-urlencoded
  - application/json

```json
{
    "url": "https://probeqa.s3.amazonaws.com/trusted_sources/video/sample-source.mp4",
    "probe_report": {
        "enabled": true,
        "diff": true
    },
    "mediainfo": {
        "enabled": true
    },
    "ffprobe": {
        "enabled": true
    }
}
```

## Report Type: **MediaInfo**

[MediaInfo](https://mediaarea.net/en/MediaInfo#google_vignette) is a convenient unified display of the most relevant technical and tag data for video and audio files. We support all the latest builds of mediainfo to ensure integrity with your production workflow.  Use the`version` tag to specify which version of mediainfo you would like run.  Use `latest`for the most recent version.

```json
  },
    "mediainfo": {
        "enabled": true,
        "version": "latest",
        "output": "JSON"
    },
```

| **Parameter** | **Description**                                   | **Allowed Values**                                        | **Default Value** |
| ------------- | ------------------------------------------------- | --------------------------------------------------------- | ----------------- |
| version       | Specifies the verion of medianinfo which will run | `latest, 24.06, 23.11, 22.12, 21.09, 20.09, 19.09, 18.12` | `latest`          |

`only`is a parameter with which you can get mediainfo response without any wrappers, meaning you get what you would get if you ran mediainfo locally. Although our defaults are usually sufficient, you can pass most mediainfo parameters. Our API parameters have the same names as mediainfo (but snake cased), so it is very easy to pass the parameters you used previously to run mediainfo. So you don't need to change your code to handle the received response we return.

## Report Type: **FFprobe**

[FFprobe](https://ffmpeg.org/ffprobe.html) is a command-line tool included with FFmpeg, primarily used for extracting and analyzing media metadata. It is is known to provide deeper introspection than media info and has the benefit of integration with ffmpeg based media processing pipelines. Chose the `only` parameter to return our default ffprobe parameters or chose to specify your custom ffprobe probe parameters, we accept virtually all ffprobe options with the same names so refer to the [ffprobe documentation](https://ffmpeg.org/ffprobe.html#Options) for details. We support all the latest builds `version` of ffprobe to ensure integrity with your production workflow.

```json
 },
    "ffprobe": {
        "enabled": true,
        "version": "latest",
        "output_format": "json"
    }
```

| **Parameter** | **Description**                                | **Allowed Values**                                                 | **Default Value** |
| ------------- | ---------------------------------------------- | ------------------------------------------------------------------ | ----------------- |
| version       | Specifies the verion of ffprobe which will run | `latest, 7.0, 6.0, 5.1, 5.0, 4.4, 4.3, 4.2, 4.1, 4.0, 3.4, or 3.3` | `latest`          |

## Report Type: **Probe Report**

The `probe_report` parses the raw JSON payloads from both ffprobe and mediainfo, structurally diffs, normalizes and canonicalizes the paramters then measurabley improves accuracy with our direct and indirect comparison algorithm.  Our algorithm has been trained with ML on over a billion media assets and improves  a vast number of incconcistencies found in a vast number diverse media assets types we have discovered managing  one of the [largest media processing pipelines](http://api.encoding.com) in the world.

```json
{,
    "inject_json": true,
    "probe_report": {
        "enabled": true,
        "diff": true
    },
```

## **Migrating from local **`ffprobe`**or  **  `mediaInfo`**commands to the Probe API** 

> _The PHP example below shows a one‑line swap: replace the local shell invocation with an HTTPS GET to the Probe API while keeping every _`ffprobe`_parameter intact._ 
>
> ```php
> // Local execution (be sure to escape vars in production)
> exec(
>     'ffprobe -i ' . escapeshellarg($url) .
>     " -print_format json -probesize $probesize" .
>     " -analyzeduration $analyzeduration -max_probe_packets $max_probe_packets" .
>     ' -show_format -show_streams',
>     $output
> );
> ```
>
> ```php
> // Equivalent call through the Probe API
> $token = getenv('PROBE_API_TOKEN');   // safer than hard‑coding
> 
> $query = http_build_query([
>     'token'                      => $token,
>     'url'                        => $url,
>     'only'                       => 'ffprobe',
>     'ffprobe[output_format]'     => 'json',
>     'ffprobe[probesize]'         => $probesize,
>     'ffprobe[analyzeduration]'   => $analyzeduration,
>     'ffprobe[max_probe_packets]' => $max_probe_packets,
>     'ffprobe[show_format]'       => 'true',
>     'ffprobe[show_streams]'      => 'true',
> ]);
> 
> $response = file_get_contents("https://api.probe.dev/v1/probe/file?$query");
> $output   = json_decode($response, true);   // handle errors in production
> ```
>
> _That’s it—you’ve migrated the command to a secure HTTPS call while preserving every original ffprobe flag._
